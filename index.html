DESKTOP SEARCH ENGINE

Introduction

One of the most frequently used and yet highly underrated features of computers is search. Now be it the default search engine of windows or spotlight on Mac, it never fails us to show the right result and we search in a lot of wide variety of terms. Be it images, text, the content of the file or file name, or file type, we have several filters and the efficiency of search engines is becoming better and better each and every day.

Now let us look at the basic python implementation of how the desktop search engine works. Of course, there will be much more complicated, faster, and more well-designed versions of this on a larger scale which are deployed in our windows and mac boxes. However, to get started with the exploration of the search system, this project will be a beginning.

Scope of the project

We will limit the search to these three things:

  1. The content of the file
  2. The file name
  3. The file type - pdf, docx, and txt

We’ll limit it to text search only. Mainly, this will be a localized search, meaning we are not going to search any term online for any of the textual content.

Note: There are advanced versions where you can search for images, and text within images. There are even advanced versions of the algorithm which is beyond the scope of this project.

Building Blocks of the project

1. Search type

The search engine will be searched with one word (unigram), two words (diagrams), and three words (trigrams). We’ll limit it to THREE words at most. We can also search based on file types. It will list all files based on file extension.

2. Preprocessing

Performs certain pre-processing techniques on the entire route directory before implementing any of the search algorithms.

Removing non-alpha characters
will remove all special characters and numbers except alphabet characters
Removing stop words
text words such as the, is, at, which, and on, all these are commonly repeated words
N-grams
the combination of 1 word, 2 words and 3 words

3. Database Architecture

We will be creating three different tables:

  • Table for the file name, file type, and file path
  • Table for words and file path: file path for every corresponding n-grams
  • Table for file creation time, file name and file path

Modules

Main.py
Creates DB_File and populates the database after the preprocessing is done. Basically, main.py puts all pieces together
DB_File.py
contains all three tables related to files and n-grams
